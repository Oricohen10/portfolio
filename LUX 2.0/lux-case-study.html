Context

Built from the ground up.

LUX 1.0 had a good seven-year run. But good enough stops being good enough.
LUX 2.0 is a full rebuild: a dedicated team, a one-year timeline, and one goal that never moved. Build a system that designers and developers genuinely own together.

What went into it
01
A three-layer token architecture
1,550 tokens. Three layers: primitives, semantics, and component scope. Every value is traceable, and a single change ripples through the entire system without touching a single file manually.
02
Figma connected to production code
Figma variables feed directly into a live pipeline. One export triggers the GitHub Action, Storybook rebuilds, and the update is live. No developer needed, no waiting.
03
Light and dark, built into the system
The first Verint product to ship dark mode. Semantic tokens make theming structural rather than cosmetic. Swap the theme and every single value follows.
04
Motion as a first-class token
Duration, easing, and animation curves live in the token layer alongside everything else. Every component pulls from the same values. Nothing is hardcoded, nothing drifts.
05
Accessibility, not an afterthought
WCAG AA compliance built in from the start. Contrast ratios, focus states, and keyboard navigation validated across both themes, before anything ships.
06
Documentation with no exceptions
Every component ships with usage guidelines, do/don't examples, a token reference, and code snippets. If the docs aren't ready, the component isn't ready.

Foundation

Every pixel traces back to one source.

Three layers. Zero hardcoded values. Every color, spacing, shadow, and radius has one origin, and one origin only.
That single source is what makes dual themes, automated pipelines, and consistent rendering across 40+ products not just possible, but reliable.

Layer 01 · Primitives
Raw values. No meaning yet.
96 tokens
The base values: colors, spacing, radii. These never appear in components directly. They only feed the layer above them.
--color-blue-500: #66AFFF --spacing-4: 16px --radius-md: 8px
Layer 02 · Semantics
Purpose, not just color.
240 tokens
These tokens describe what a value is used for, not what it is. They reference primitives and define meaning. Light and dark themes are defined at this layer.
--color-interactive-default: var(--color-blue-500) --color-surface-raised: var(--color-gray-900)
Layer 03 · Component Tokens
Scoped to each component.
1,113 tokens
Each component has its own token namespace. Changing a button's background color has no effect on anything else in the system.
--button-bg-primary: var(--color-interactive-default) --input-border-focus: var(--color-interactive-default)
primitives.css → semanticsLight.ts → button.style.ts 
/* 01 - primitives.css */

The Pipeline

Zero terminal.
Designer to GitHub in 60 seconds.

Most token pipelines need a developer to run a build script, commit the output, and open a PR by hand. This one runs entirely in the browser, edit a Figma variable, click once in the plugin, and a validated pull request lands on GitHub automatically. No terminal. No build step. No waiting.

Tooling

Built to close the gaps.

No off-the-shelf plugin fit what LUX 2.0 needed.
So we built four of them, each one eliminating a different manual step from the workflow.
Less friction, more time designing.

Layout Scaffolding
Layout Creator
Scaffold a full LUX 2.0 layout in seconds. Generates containers, navigation, and page structure with the correct token bindings already applied.
Data Table Generator
Table Creator
Generate data tables with configurable rows, columns, and content. Every output matches the LUX 2.0 Data Table token structure exactly.
Token Pipeline Automation
Token Sync
Pushes updated design tokens from Figma directly into the pipeline. One click and tokens.json commits, the GitHub Action fires, and Storybook rebuilds.
Icons Search
Icon Library Picker
Browse and insert icons from the LUX 2.0 icon library. Search by name, category, or meaning. Keeps icon usage consistent across every Figma file.
Documentation

Documented without exception.

Every component ships with the full picture: usage guidelines, visual do/don't examples, a token reference, accessibility notes, and code snippets.
Simple rule: docs not done, component not done.
